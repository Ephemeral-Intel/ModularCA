# ModularCA

ModularCA is a certificate authority written in C# that aims to provide common CA features in a free package. The current focus is implementing ACME support so internal services can renew their certificates automatically. The web frontend is written in React with TypeScript.

> **Status:** Active development — not ready for production use. There is no production release yet; all current code is in testing, and the project is currently maintained by a single person. The speed and feasibility of a release depend on contributions. The goal is to provide a strong foundation to attract further development support.

## Features

#### ACME Protocol Support
ModularCA supports creating ACME endpoints for each CA endpoint.

#### CA URL Endpoint Generation
Each CA generated has a URL endpoint from which to retrieve the CA itself and its CRLs. An HTTP frontend webpage serves details on the hosted CAs and their CRLs.

#### Functionality-Focused
Designed around the functionality the average administrator will appreciate — from quotas to tenancy, ModularCA focuses on providing the core feature set.

## Current Features
- Simple certificate request and signing via the development API.
- Bootstrap a MySQL/MariaDB database with tables and necessary entries.
- Generate a system CA cert for internal functions and file signing.
- Generate a default CA cert for all certificate signing functions.
- Generate a custom keystore that signs all entries with the system and default CA cert for integrity.
- Double-sign the final custom keystore file for additional integrity.
- Store the keystore password in two parts — half in the database, half in a file.
- Frontend administrator, user, and public pages.

## Secure By Design Principles
A number of features are aimed at a secure-by-design format:
- In-memory private key processing without storing the plain private key in the filesystem.
- Two-layer keystore unlocking process, increasing the work and access a threat actor needs.
- Private keys generated by the program for a certificate signing request are encrypted with the system certificate.
- Private keys that are part of a signed certificate are decrypted and re-encrypted with the signing CA certificate.
- Keystore anti-tampering via entry and file signing.
- MFA for all sensitive operations.
- Configurable ceremonies for high-impact CA actions.

Planned for exploration:
- Keystore loading only when in use, with unloading after use.
- Database connection creation on use and destruction after use.

## Build

#### Requirements
- **.NET 10** for the API backend — https://learn.microsoft.com/en-us/dotnet/core/install/
- **dotnet ef migrations** tool (after .NET 10 is installed): `dotnet tool install --global dotnet-ef`
- **npm + Node.js** for the web frontend — https://nodejs.org/en/download
- **MySQL/MariaDB** for the database backend — https://mariadb.org/download/

1. Clone the repository
```
git clone https://github.com/Ephemeral-Intel/ModularCA.git
cd ModularCA
```

2. Restore and build the application
```
dotnet restore
dotnet build ModularCA.API/ModularCA.API.csproj -c Development   # or Staging / Production
```

3. Run
```
dotnet run --project ModularCA.API -c Development   # or Staging / Production
```

##### Operational commands (run a task, then exit)

| Flag | Arguments | Description |
|------|-----------|-------------|
| `--bootstrap` | — | Provisions a fresh install: database, tenants, groups, roles, profiles/OIDs, the system + default CA certs, and the double-signed keystore. The CLI equivalent of the web setup wizard. |
| `--reset --force` | — | **Destructive.** Factory reset — drops databases, keystores, and config so the install can be bootstrapped again. `--force` is mandatory. The audit database is *not* dropped (retained by design), and `config/backup.key` is renamed to `backup.key.pre-reset-<ts>`. |
| `--confirm-db-name <name>` | DB name | Required with `--reset --force` — the operator must type the target database name to prove they know what is being destroyed. |
| `--expected-db <name>` | DB name | Backwards-compatible alias for `--confirm-db-name`. |
| `--ci-no-confirm` | — | Waives the `--confirm-db-name` gate for unattended CI pipelines that genuinely need an automated reset. |
| `--backup [<path>]` | output path (optional) | Writes an encrypted backup archive. Defaults to a built-in location if no path is given. |
| `--restore <archive>` | archive path | Restores from an encrypted backup archive. |
| `--password <pwd>` | password | Used with `--restore` for `StoredPassword`-mode archives when the local password key file is missing or doesn't match. If omitted when required, the archive password is prompted for interactively. |

##### Runtime flags (modify a normal launch)

| Flag | Arguments | Description |
|------|-----------|-------------|
| `--setup-local` | — | Allows the setup wizard to accept connections from RFC 1918 private networks. |
| `--setup-bind <address>` | IP address | Binds the setup wizard listener to a specific address. Defaults to `0.0.0.0`; an invalid value falls back to `0.0.0.0` with a warning. A loud warning is emitted if the bind lands on a non-RFC1918 interface. |
| `--allow-dev-mode` | — | Required to start when `ASPNETCORE_ENVIRONMENT=Development`, since the developer exception page can leak stack traces, file paths, and SQL. Equivalent to setting `MODULARCA_ALLOW_DEV_MODE=1`. |

##### Examples

```bash
# Provision a fresh install
dotnet run --project ModularCA.API -- --bootstrap

# Factory reset (destroys all data; audit DB is retained)
dotnet run --project ModularCA.API -- --reset --force --confirm-db-name modularca

# Unattended reset in CI
dotnet run --project ModularCA.API -- --reset --force --ci-no-confirm

# Back up and restore
dotnet run --project ModularCA.API -- --backup ./backups/modularca.enc
dotnet run --project ModularCA.API -- --restore ./backups/modularca.enc
```

#### Installation Notes
Development builds are created at an arbitrary rate, determined by the changes made. Once a development build is created, it is published as a release. No release of ModularCA will be considered production-ready until the frontend functionality is complete.

## Reasoning
There is a low number of open-source CA projects. Many of the best CA features are only available in commercial or paid-subscription products. By developing a new application with those features free in mind, ModularCA aims to provide a product that can compare with commercial solutions.
